# お出かけ準備 ＆ 思い出ログアプリ（パック＆メモ）仕様書

本ドキュメントは、モバイル向けお出かけ準備 ＆ 思い出ログアプリ（仮称：パック＆メモ）のシステム仕様および設計方針を定義します。

---

## 1. 概要

「パック＆メモ」は、お出かけ前の「準備（持ち物チェック）」と、お出かけ当日の「思い出記録（写真＋一言メモ）」をイベント単位で一元管理できるパーソナルアシスタントアプリです。

### 1.1 開発ターゲット
*   **動作環境**: モバイル (Android)
*   **検証端末**: Pixel 10 Pro
*   **将来的な展望**: iOS（iPhone）での動作も想定したクロスプラットフォーム開発

---

## 2. 技術スタックの選定

将来的なiOSへの移植性と、Pixel 10 Proでのネイティブパフォーマンスを両立するため、技術スタックには**React Native (Expo)**を採用します。

### 採用技術スタック：React Native (Expo)
*   **採用理由**:
    *   **高い移植性**: Android/iOSの両環境において、TypeScript / React を使用してシングルコードベースから高品質なネイティブUIを迅速に構築可能。
    *   **Expoによる強固なエコシステム**: カメラ・写真ライブラリへのアクセスやファイルシステムの操作に標準化されたExpo SDK（`expo-image-picker`等）が利用でき、実機検証（Expo Go）が非常に容易。
    *   **最新OS・ネイティブ機能のサポート**: Android（Pixel 10 Pro）の新しいフォトピッカーや、Edge-to-Edge（エッジ・トゥ・エッジ）表示に標準対応。
*   **主要ライブラリ**:
    *   **UI / 状態管理**: React Native SDK ＋ Expo Router（ファイルベースルーティング） ＋ React Context API
    *   **ローカルデータベース**: `expo-sqlite`（軽量で高速、モバイルに最適化されたSQLiteデータベース）
    *   **カメラ/写真連携**: `expo-image-picker`（最新のシステムフォトピッカーやカメラ起動への安全なアクセス）
    *   **ファイルシステム**: `expo-file-system`（アプリ専用のドキュメントディレクトリへのファイル保存）

## 3. 画面構成と機能要件

アプリは主に以下の3つの画面で構成されます。

### 3.1 ホーム画面 (Home Screen)
*   **機能**: 登録されているお出かけイベントの一覧表示、新規イベントの作成・削除。
*   **UI要素**:
    *   イベントカードリスト（イベント名、お出かけ予定日、準備状況の進捗インジケータなどを表示）
    *   「新規イベント追加」フローティングアクションボタン (FAB)
*   **インタラクション**:
    *   カードをタップ：対象イベントの「準備画面」へ遷移。
    *   カードを長押し、またはスワイプ：イベントの編集/削除メニュー。

### 3.2 準備画面 (Checklist Screen)
*   **機能**: 選択したイベントの持ち物リストを表示、チェック状態の管理、新規持ち物の追加。
*   **UI要素**:
    *   イベント名と日付のヘッダー表示
    *   持ち物リスト（チェックボックス、持ち物名、削除ボタン）
    *   「持ち物を追加」入力フォーム（テキスト入力 ＋ 追加ボタン）
    *   「思い出を記録する」ボタン（思い出画面へ遷移）
*   **インタラクション**:
    *   チェックボックスのタップ：`isChecked` 状態を反転し、進捗状況を更新。
    *   「持ち物を追加」：リストの末尾に新規アイテムを追加。

### 3.3 思い出画面 (Memory Screen)
*   **機能**: お出かけ当日または終了後に、思い出の写真1枚と一言メモを記録。
*   **UI要素**:
    *   写真表示エリア（未選択時はプレースホルダー、選択時はプレビュー表示）
    *   「写真を撮る（カメラ）」ボタン ＆ 「アルバムから選ぶ」ボタン
    *   一言メモ入力エリア（複数行テキスト入力、最大140文字程度）
    *   「保存」ボタン
*   **Pixel 10 Pro 固有の考慮点**:
    *   Android 14/15の新しいフォトピッカー（Photo Picker）APIを使用して、ユーザーが必要な画像のみに安全にアクセスできるように権限を管理する。
    *   高解像度カメラで撮影された大容量画像の圧縮処理を行い、アプリ内のローカル領域（App Document Directory）に最適化して保存する。

#### 3.4.2 画面遷移仕様一覧

| 遷移元画面 | 遷移先画面 | トリガー (ユーザー操作) | データの受け渡し / 状態変化 | 備考 |
| :--- | :--- | :--- | :--- | :--- |
| **ホーム画面** | **新規イベント追加** | 画面右下の「+」ボタン (FAB) をタップ | なし。シートが下部からスライドイン。 | ボトムシート形式で表示。背景タップで閉じる。 |
| **新規イベント追加** | **ホーム画面** | ①「作成」タップ<br>②「キャンセル」または背景タップ | ① `TripEvent` インスタンスを新規生成しローカルDBへ保存、一覧を再描画。<br>② データの変更を行わず、シートを閉じる。 | バリデーション（イベント名未入力など）が通らない場合は作成不可。 |
| **ホーム画面** | **準備画面** | イベントリスト内の任意のカードをタップ | タップされた `TripEvent.id` を遷移先パラメータとして渡す。 | 該当イベントに紐づく持ち物リストが描画される。 |
| **準備画面** | **ホーム画面** | 左上「戻る」ボタン（矢印アイコン）または端末の戻るジェスチャー | なし。 | 持ち物チェックの変更はタップ時に随時ローカルDBへ自動保存されるため、警告なしで戻る。 |
| **準備画面** | **思い出画面** | 画面下部の「思い出を記録する」ボタンをタップ | `TripEvent.id` を遷移先パラメータとして渡す。 | 既に思い出が保存されている場合は、入力済みの写真とメモが初期表示される。 |
| **思い出画面** | **準備画面** | ①「保存」ボタンをタップ<br>②「戻る」ボタン（変更なし時） | ① `EventMemory` データを更新しローカルDBに保存後、遷移。<br>② データの変更を行わず、そのまま準備画面へ戻る。 | 変更がある状態で「戻る」場合は後述の例外フロー（未保存警告）へ。 |
| **思い出画面** | **システムカメラ / フォトピッカー** | 「写真を撮る」または「アルバムから選ぶ」をタップ | インテント（カメラ / フォトピッカー）の起動。 | Pixel 10 Pro では、Android 14/15 準拠の新しい Photo Picker API を優先利用。 |
| **システムカメラ / フォトピッカー** | **思い出画面** | 写真の選択完了、またはカメラ撮影完了 | 選択・撮影された画像の一時保存パスを思い出画面に返し、画面上にプレビュー表示。 | この時点ではまだDB保存は未完了。ユーザーが「保存」をタップして初めて本保存される。 |

#### 3.4.3 例外・エラー時のコントロールフロー

1.  **思い出画面における「未保存のまま戻る」 (E-1)**
    *   **発生条件**: 思い出画面で「写真の新規選択/変更」または「一言メモの入力/編集」を行った状態で、保存ボタンを押さずに「戻る（またはシステム戻るジェスチャー）」を操作した場合。
    *   **挙動**: 標準確認ダイアログ（「変更が保存されていません。破棄して戻りますか？」）を表示。
        *   「編集を続ける」を選択 ➔ ダイアログを閉じ、思い出画面の入力を継続。
        *   「破棄して戻る」を選択 ➔ 入力を破棄し、ローカルDBを更新せずに準備画面へ遷移。
2.  **カメラ起動中、フォトピッカー選択中のOSによるアクティビティ破棄 (E-2)**
    *   **想定事象**: メモリ負荷等により、カメラ撮影中にバックグラウンドのアプリプロセスがOSにより一時的に破棄されるケース。
    *   **対策**: Flutter の `WidgetsBindingObserver` や Android の `onActivityResult` のリカバリー機構を考慮。アプリ復帰時に、撮影完了した画像が正しく処理されるよう状態復旧設計を行う。
3.  **ストレージ容量不足による画像保存失敗 (E-3)**
    *   **想定事象**: カメラで撮影した高解像度画像をローカルドキュメント領域へコピーしようとした際、端末ストレージに空きがない場合。
    *   **対策**: 画像コピー（保存）処理を `try-catch` で囲み、失敗時はスナックバーまたはダイアログで「ストレージ空き容量が不足しているため、画像を保存できませんでした」とエラーメッセージを表示し、入力を維持する。

## 4. データ構造とデータモデル

イベントおよびそれに紐づく持ち物リスト、思い出ログは以下のスキーマ（JSONイメージ）で定義します。

### 4.1 データモデル定義 (TypeScript風表記)

```typescript
interface PackingItem {
  id: string;          // 持ち物アイテムのユニークID
  name: string;        // 持ち物名 (例: "お弁当", "レジャーシート")
  isChecked: boolean;  // チェック状態
}

interface EventMemory {
  photoUri: string | null;  // 写真のローカル保存パス (例: "file:///data/user/0/.../image.jpg")
  note: string;             // 一言メモ (最大140文字)
}

interface TripEvent {
  id: string;               // イベントのユニークID
  title: string;            // イベント名 (例: "週末の動物園")
  date: string;             // お出かけ予定日 (YYYY-MM-DD)
  items: PackingItem[];     // 持ち物チェックリスト
  memory: EventMemory;      // 思い出ログ
}
```

### 4.2 保存方針 (Local Storage)
*   **データ永続化**: `expo-sqlite` を利用し、モバイル端末のセキュアな内部ストレージ領域にSQLiteデータベースファイルとして保存します。
*   **写真データ**: カメラやフォトピッカーから取得した画像ファイルは、システムの一時領域（キャッシュ）からアプリ専用のドキュメントディレクトリ（`expo-file-system` の `documentDirectory` 内）へコピーして一意のファイル名（例: `memories/${eventId}_memory.jpg`）で保存し、データベースにはその相対パスを記録します。これにより、OSによるキャッシュクリーンアップでの消失を防ぎ、iOSのサンドボックスのパス変更（UUID動的生成）による読み込み失敗を防ぎます。

## 5. ユーザビリティとテスト検証計画

### 5.1 Pixel 10 Pro（Android 15以降想定）での検証項目
1.  **エッジトゥエッジ（Edge-to-Edge）表示の対応**: ナビゲーションバーやステータスバーとUI要素が重ならないこと。
2.  **フォトピッカー連携**: メディア権限を過剰に要求せず、システムの標準フォトピッカー経由で画像が正しくインポートできること。
3.  **カメラ連携**: カメラアプリ起動から撮影、写真切り出し、アプリ復帰までの一連のライフサイクルが安定して動作すること（メモリ不足によるアプリの強制終了対策）。
4.  **画面回転時の状態保持**: 思い出画面のテキスト入力中などに端末を回転させても、入力内容やプレビュー画像が消失しないこと。

### 5.2 将来のiOS展開時の考慮事項
*   **ファイルパス**: iOSのサンドボックス構造上、アプリ再起動時にファイルの絶対パスが変わる可能性があるため、保存時には相対パス（ドキュメントディレクトリからの相対位置）で管理し、読込時にベースパスを結合する設計にすること。
