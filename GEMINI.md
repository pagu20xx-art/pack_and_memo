## 🔍 概要
Pack and Memo は、パーソナルなパッキング（荷造り）アシスタント、または生産性向上アプリケーションです。

## 🛠️ 技術スタック
- **フレームワーク**: React Native (Expo)
- **状態管理**: React Context API
- **データ永続化**: `expo-sqlite` (SQLite) ＆ `expo-file-system` (写真などのファイル保存)
- **カメラ・写真連携**: `expo-image-picker`
- **言語**: TypeScript

---

## 🏗️ アーキテクチャと設計原則

高い技術的整合性とクリーンなコードを維持するため、すべてのコントリビューター（AIアシスタントを含む）は以下の原則に従う必要があります。

1. **継承よりも合成（Composition Over Inheritance）を優先**: 複雑な継承よりも、明示的な合成や委譲（例：ラッパークラス、プロキシ、ファクトリ関数など）を優先します。
2. **明示的かつイディオマティックなコード**: 警告の非表示化/抑制や、型システムのバイパス（例：TypeScriptでの不要な `any` キャストなど）といったハックは避けてください。型ガードや明示的なインターフェース実装など、言語の明示的な機能を活用します。
3. **サージカル（局所的）な修正**: 既存のモジュールを編集する際は、要求された変更のみに集中してください。明示的に要求されない限り、無関係なリファクタリングや外部コードの「クリーンアップ」は避けてください。
4. **洗練されたビジュアル（Visual Polish）**: UIが含まれる場合は、一貫したスペース、磨き上げられたインタラクティブ状態、および適切なレイアウトにより、高品質なビジュアルを保証してください。

## 🚀 開発ガイドライン

### 1. コードスタイルとリンター
- 標準的なフォーマットおよびリンティングルールに従います（今後設定予定）。
- 変更をコミットする前に、必ずリンターとフォーマッター（`prettier`、`eslint` など）を実行してください。

### 2. テスト戦略
- **経験的検証（Empirical Validation）**: バグ修正の際は、不具合を再現するテストケースまたは検証スクリプトを必ず同封してください。
- **包括的なカバレッジ**: 新機能を追加する際は、正常系と異常系の両方を検証する自動テストを含める必要があります。
